# v0.2 instance expansion

The v0.2 research task contains **25 concepts, 100 seeded instances, and 200 paired samples**. The
200 samples are arranged as 100 clean/perturbed pairs: each of 25 reviewed family contracts is
evaluated under four deterministic scenario seeds. This is replication, not 200 independent
evaluation concepts.

`data/task_specs/v0.2-instances.json` is the machine-readable catalog. It records the stable instance
ID, family ID, v0.2 scoring-contract version, inherited family-spec version, seed, category,
difficulty, declared workflow steps, optimal tool calls, enforced dependency depth, horizon-claim
status, prompt, clean sample ID, perturbed sample ID, and perturbation. It is
regenerated with:

```bash
decision-agent-bench export-instances data/task_specs/v0.2-instances.json
```

The four perturbed instances cycle deterministically through each family's perturbation list. This
activates all 53 named perturbations at least once across the 100 perturbed samples, rather than
selecting only the first perturbation from each family.

## Why expand by scenario seed

Repeating one fixed database risks measuring memorization of entity values rather than stable
decision behavior. The four seeds preserve designed structural facts—such as the R03 decline,
P003 recall, C00001 refund cluster, and top-three P001 allocation stores—while varying non-critical
transaction counts and forecasts. Tests verify those answer contracts for every published seed.

The expansion increases statistical replication without pretending that seeded variations are
independent task concepts. Many answer keys intentionally remain fixed across the four seeds while
nuisance values vary. Results must cluster or pair by concept and seed where appropriate. The
original v0.1 25-family/50-sample task remains registered so historical results do not silently
change.

## Horizon claim boundary

The frozen family specs contain a field historically named `horizon`. v0.2.1 publishes it as
`declared_workflow_steps`, alongside the measured `optimal_tool_calls` and
`enforced_dependency_depth`. The median values are 12 declared steps, four optimal tool calls, and
zero enforced dependent state transitions. The expanded category formerly labeled
`long_horizon_workflow` is therefore published as `workflow_planning`. See
[horizon methodology](horizon-methodology.md).

## Versioned decision utility

v0.2 adds an information-matched replacement-product oracle for `DAB-ASS-001`. It exhaustively
enumerates active same-category products whose vendor can satisfy its minimum order, computes each
candidate's observed 28-day unit-margin opportunity at S001, and reports normalized regret against
the best feasible candidate. The original v0.1 task retains its frozen effectiveness-based decision
quality; experiment configuration rejects v0.2 runs mislabeled with the v0.1 scoring version.

`configs/experiments/v0.2.template.json` defines the complete eight-baseline, two-variant,
three-repetition architecture and ablation grid. Run `estimate-experiment` before enabling provider
models: one enabled model requires 4,800 sample executions, and three require 14,400.
